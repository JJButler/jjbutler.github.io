This document describes the different APIs exposed as RESTful web services

Account API

  • Description: this endpoint allows clients to create or update accounts in Sciba
  • URL endpoint: /ws/division/$division/account/$account where
    • $division: a string that identifies the division code in Sciba
    • $account: a string that identifies the account code in Sciba
  • Method: POST
  • Parameters: the json body request can specify the following parameters:
Name Type Required? Description
accountType String YES Account type to create or update, valid values are:
CLIENT_CORP: for client corporate accounts
CLIENT_PERSON: for client person accounts
COMMISSION_AGENT: for commission agent accounts
THIRD_PARTY_CORP: for third-party corporate accounts
THIRD_PARTY_PERSON: for third-party person accounts
controlAccount String representing a Sciba control account code NO Control account to link against the account
addresses List of addresses where each address is specified by: NO List of addresses associated to the account. NB: if the account already exists and a list of addresses is specified the current addresses in Sciba will be deleted and updated with the new list.
-address1: String YES First line of address
-address2: String NO Second line of address
-address3: String NO Third line of address
-city: String NO City name
-county: String NO County name
-country: String NO Country name
-postcode: String YES Pstal code
bankAccounts List of bank accounts where each bank account is specified by: NO List of bank accounts linked to the account. NB: if the account already exists and a list of bank accounts is provided in the request the current bank account list in Sciba will be deleted and updated with the new list.
-name: String YES Account’s name
-sortCode: String YES Account’s sortcode
-accountNumber: String YES Account’s number
-description: String NO Account’s description
-bank: String NO Bank’s name
name String YES Name for the account in Sciba
email String NO Email associated to the account
title String NO Account’s holder title. NB only applies to ‘CLIENT_PERSON’ account types
firstName String NO Account’s firstname. NB only applies to ‘CLIENT_PERSON’ account types
surname String NO Account’s surname. NB only applies to ‘CLIENT_PERSON’ account types
dob Date NO Account’s dob. NB only applies to ‘CLIENT_PERSON’ account types
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 1978-01-08T00:00:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 1978-01-08T00:00:00Z)
  • Request example:
    • Request: http://localhost:8080/ws/division/T-ST/account/TST0001
    • Method: POST
    • Body:
      {
    "accountType": "CLIENT_PERSON",
    "addresses": [
    {
        "address1": "1, Dizzy Heights",
        "address2": "The Avenue",
        "address3": "Central District",
        "city": "Somewhereville",
        "county": "Gloucestershire",
        "country": "United Kingdom",
        "postcode": "ST1 2AB"
    },
    {
        "address1": "2, Othertown",
        "postcode": "OT2 4AB"
    }
    ],
    "bankAccounts": [
    {
        "name": "PP BA 1",
        "description": "Primary bank account",
        "bank": "Santander",
        "accountNumber": "11111111",
        "sortCode": "01-02-03"
    },
    {
        "name": "PP BA 2",
        "accountNumber": "22222222",
        "sortCode": "02-03-04"
    }
    ],
    "dob": "1978-01-08T00:00:00Z",
    "email": "jb@jltgroup.com",
    "firstName": "Jeremy",
    "name": "Mr J Butler",
    "surname": "Butler",
    "title": "Mr"
}
    • Response example:
      {
    "_links": {
    "self": {
        "href": "/account/show/100001"
    }
    }
}

Product API

  • Description: this endpoint allows clients to create or update products in Sciba
  • URL endpoint: /ws/division/$division/product/$productCode/version where
    • $division: a string that identifies the division code in Sciba
    • $productCode: a string that identifies the product code in Sciba
  • Method: POST
  • Parameters: the json body request can specify the following parameters:
Name Type Required? Description
name String YES Product version name
policySections List of policy sections where each policy section is defined by: YES List of policy sections that define the product.
-name: String YES Policy section name
-covers: list of covers where each cover is defined by: YES
-name: String YES Cover’s name
-binderSection: binder for that cover defined by: YES
-binder: String YES Binder’s name
-name: String YES Binder’s section name
  • Request example:
    • Request: http://localhost:8080/ws/division/T-ST/product/PTG/version
    • Method: POST
    • Body:
      {
    "name": "1.0",
    "policySections": [
    {
        "name": "StructureAndAccessories",
        "covers": [
        {
            "name": "Structure",
            "binderSection": {
            "binder": "B10090039/12",
            "name": "Material Damage"
            }
        },
        {
            "name": "Accessories",
            "binderSection": {
            "binder": "B10090039/12",
            "name": "Material Damage"
            }
        }
        ]
    },
    {
        "name": "Contents",
        "covers": [
        {
            "name": "Contents",
            "binderSection": {
            "binder": "B10090039/12",
            "name": "Material Damage"
            }
        }
        ]
    }
    ]
}
    • Response example:
      {
    "_links": {
    "self": {
        "href": "/productVersion/show/1"
    }
    }
}

Policy API

  • Description: this endpoint allows clients to create or update policies in Sciba
  • URL endpoint: /ws/division/$division/account/$account/policy/$policy where
    • $division: a string that identifies the division code in Sciba
    • $account: a string that identifies the account code in Sciba
    • $policy: a string that identifies the policy reference in Sciba
  • Method: POST
  • Parameters: the json body request can specify the following parameters:
Name Type Required? Description
lpn String YES Long Policy Number which helps to track renewals for a policy
inceptionDate Date YES Policy’s inception date.
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 2014-01-08T00:00:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 2014-01-08T00:00:00Z)
expiryDate Date YES Policy’s expiry date.
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 2015-01-08T00:00:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 2015-01-08T00:00:00Z)
productVersion JSON object which specifies: YES Links the policy to a specific product version in Sciba
-product: String YES Product’s name
-name: String YES Version’s name
renewalCount Integer YES Identifies whether the policy is a NB (renewalCount == 0) or a RN (renewalCount > 0)
  • Request example:
    • Request: http://localhost:8080/ws/division/T-ST/account/TST0001/policy/POLICY0001
    • Method: POST
    • Body:
      {
    "lpn": "PTS000001",
    "inceptionDate": "2012-12-07T15:00:00.0Z",
    "expiryDate": "2013-12-07T00:00:00.0Z",
    "renewalCount": 0,
    "productVersion": {
    "product": "RET",
    "name": "1"
    }
}
    • Response example:
      {
    "_links": {
    "self": {
        "href": "/policy/show/100001"
    }
    }
}

Policy Event API

  • Description: this endpoint allows clients to create policy events in Sciba
  • URL endpoint: /ws/division/$division/account/$account/policy/$policy/policyEvent/$policyEvent where
    • $division: a string that identifies the division code in Sciba
    • $account: a string that identifies the account code in Sciba
    • $policy: a string that identifies the policy reference in Sciba
    • $policyEvent: a string that identifies the policy event reference in Sciba
  • Method: POST
  • Parameters: the json body request can specify the following parameters:
Name Type Required? Description
type String YES Police event type, valid values are: NB (New Business), RN (Renewal), MTA (MTA), CNCL (Cancellation)
effectiveDate Date YES Policy event effective date.
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 2014-01-08T00:00:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 2014-01-08T00:00:00Z)
remittanceDetails String YES Key-value pairs separate by ampersand which are used for remittance notification
transaction JSON object which specifies: YES
-methodOfPayment: String YES Transaction’s method of payment. Valid values are: CASH (Cash), DEBIT_CARD (Card - Debit), CREDIT_CARD (Card - Credit), GCARD (Card - Unknow), CHEQUE (Cheque), BACS (BACS), DIRECT_DEBIT (Direct Debit), POSTAL_ORDER (Postal Order)
premium JSON object which specifies: YES Gives full details of the premium breakdown, i.e. breakdown per cover’s product and type including what is insured under which cover
-totalGrossPremium: Integer YES Amount in pence for the total premium (gross)
-totalIPT: Integer YES Amount in pence for the total IPT
-totalGrossPremiumPlusIPT: Integer YES Amount in pence for the total gross + IPT
-totalChargeable: Integer YES Amount in pence for the total chargeable (inc. fees)
-elements: array to provide a detailed breakdown of the chargeable amount as: YES
+$id: Integer YES/NO Required for IPT type elements which identifies the tax element. Ignored for any other element type.
+taxed: JSON object which links element’s amount to tax element when the element is liable to IPT YES
-$ref: Integer YES Required for taxable elements and refers to the id of the tax element for the same cover
+amount: Integer YES Amount in pence for this element
+type: String YES Identifies the element’s type: ‘Underwriter Net Premium’, ‘Our Commission’, ‘IPT’
+cover: JSON object which gives information about the element’s cover by specifying: YES
-policySection: String YES Product’s policy section name
-name: String YES Product’s cover name
-insured: array to provide a detailed breakdown of the sum insured as: YES
+sumInsured: Integer YES Sum insured amount in POUNDS
+excess: Integer YES Excess amount in POUNDS
+cover: JSON object which gives information about the insured element’s cover by specifying: YES
-policySection: String YES Product’s policy section name
-name: String YES Product’s cover name
  • Request example:
    • Request: http://localhost:8080/ws/division/T-ST/account/TST0001/policy/POLICY0001/policy-event/EVENT0001
    • Method: POST
    • Body:
      {
    "type": "NB",
    "effectiveDate": "2012-12-07T15:00:00.0Z",
    "remittanceDetails": "templateID=CLM001&param1=50.00&param2=100",
    "transaction": {
    "methodOfPayment": "Cash"
    },
    "premium": {
    "totalGrossPremium": 32.08,
    "totalIPT": 1.92,
    "totalGrossPremiumPlusIPT": 34,
    "totalChargeable": 46.5,
    "elements": [
        {
        "$id": "1",
        "amount": 1.92,
        "type": "IPT",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        }
        },
        {
        "taxed": {
            "$ref": "1"
        },
        "amount": 7.86,
        "type": "Underwriter Net Premium",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        }
        },
        {
        "taxed": {
            "$ref": "1"
        },
        "amount": 0.57,
        "type": "Payaway Commission",
        "ibaAccountRef": "AGT0001",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        }
        },
        {
        "taxed": {
            "$ref": "1"
        },
        "amount": 23.65,
        "type": "Our Commission",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        }
        },
        {
        "amount": 12.5,
        "type": "Our Admin Fee"
        }
    ],
    "insured": [
        {
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        },

        "excess": 50
        }
    ]
    }
}

      {

Binder API

  • URL endpoint: /ws/division/$division/binder/$binder where
    • $division: a string that identifies the division code in Sciba
    • $binder: a string that identifies the binder code in Sciba
  • Method: PUT
  • Parameters: the json body request can specify the following parameters:
Name Type Required? Description
startDate Date YES Binder’s start date
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 2015-01-08T00:00:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 2015-01-08T00:00:00Z)
endDate Date YES Binder’s end date
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 2015-01-08T00:00:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 2015-01-08T00:00:00Z)
year Integer YES Binder’s year
  • Request example:
    • Request: http://localhost:8080/ws/division/T-ST/binder/TST2014
    • Method: PUT
    • Body:
      {
    "startDate": "2012-01-01T00:00:00.0Z",
    "endDate": "2012-12-31T00:00:00.0Z",
    "year": 2012
}
    • Response example:
      {
    "_links": {
    "self": {

    }
    }
}

Claim API

Insert/Update Claim

  • Description: this endpoint allows clients to create or update claims in Sciba
  • URL endpoint: /ws/division/$division/policy/$policy/claim/$claim where
    • $division: a string that identifies the division code in Sciba
    • $policy: a string that identifies the policy reference in Sciba
    • $claim: a string that identifies the claim reference in Sciba
  • Method: PUT
  • Parameters: the json body request can specify the following parameters:
Name Type Required? Description
dateOfLoss Date YES Claim’s date of loss
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 2014-04-08T12:23:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 2014-04-08T12:23:00Z)
claimSummary String YES Claim’s summary
claimDetails String NO Claim’s details
insured Boolean NO
totalLoss Boolean NO
state String YES Claim’s state, valid values are: ‘OPEN’ and ‘CLOSED’

  • NB: if the claim number $claim already exists in Sciba the claim record will be updated, so the current claim details will be overwritten with the new ones. Therefore, when using this endpoint it is advisable to always provide all the available details.

  • Request example:

    • Request: http://localhost:8080/ws/division/T-ST/policy/POLICY0001/claim/CLAIM0001
    • Method: PUT
    • Body:
      {
    "dateOfLoss": "2013-01-01T00:00:00.0Z",
    "claimSummary": "Camera stolen",
    "claimDetails": "Taken from back of chair while eating in restaurant.",
    "insured": true,
    "state": "OPEN"
}
    • Response example:
      {
    "_links": {
    "self": {
        "href": "/claim/show/100001"
    }
    }
}

Settlement

  • Description: this endpoint allows clients to create settlements for claims in Sciba
  • URL endpoint: /ws/division/$division/policy/$policy/claim/$claim/settlement where
    • $division: a string that identifies the division code in Sciba
    • $policy: a string that identifies the policy reference in Sciba
    • $claim: a string that identifies the claim reference in Sciba
  • Method: POST
  • Parameters: the json body request can specify the following parameters:
Name Type Required? Description
date Date YES Claim event effective date.
Supported date formats are:
-yyyy-MM-dd’T’HH:mm:ss.SSSSSS’Z’ (example: 2014-01-08T00:00:00.000000Z)
-yyyy-MM-dd’T’HH:mm:ss’Z’ (example: 2014-01-08T00:00:00Z)
reservedAmounts Array of JSON objects where each element specifies: YES List of reserve amounts for this claim event
-amount: Integer YES Reserve amount in pence
-fees: Integer YES Fee amount in pence
-subcategory: String NO Reserve subcategory name
-cover: JSON object which gives information about the reserved amount cover by specifying: YES
-policySection: String YES Product’s policy section name
-name: String YES Product’s cover name
settlement JSON object which specifies: YES Gives full details of the settlement breakdown, i.e. breakdown per cover’s product and type
-payeeAccountRef: String YES Payee account reference (i.e. account code in Sciba)
-invoiceRef: String YES Invoice reference number
-exGratia: Boolean YES
-insurerPaid: Boolean YES
-payNetOfOutstandingLoan: Boolean YES
-elements: array to provide a detailed breakdown of the settlement as: YES
+$id: Integer YES/NO Required for VAT type elements which identifies the tax element. Ignored for any other element type.
+taxed: JSON object which links element’s amount to tax element when the element is liable to IPT YES
-$ref: Integer YES Required for taxable elements and refers to the id of the tax element for the same cover
+amount: Integer YES Amount in pence for this element
+type: String YES Identifies the element’s type: ‘Amount’, ‘Fees’, ‘Excess’, ‘VAT’
+subcategory: String YES Settlement subcategory name
+cover: JSON object which gives information about the element’s cover by specifying: YES
-policySection: String YES Product’s policy section name
-name: String YES Product’s cover name
  • Request example:
    • Request: http://localhost:8080/ws/division/T-ST/policy/POLICY0001/claim/CLAIM0001/settlement
    • Method: POST
    • Body:
      {
    "date": "2013-01-08T09:11:52.0Z",
    "reserveAmounts": [
    {
        "amount": 400,
        "fees": 80,
        "cover": {
        "policySection": "Equipment",
        "name": "Equipment"
        },
        "subcategory": null
    }
    ],
    "settlement": {
    "payeeAccountRef": "CLT0003",
    "invoiceRef": "REPAIR0001",
    "exGratia": false,
    "insurerPaid": false,
    "payNetOfOutstandingLoan": false,
    "elements": [
        {
        "$id": "1",
        "amount": 20,
        "type": "VAT",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        },
        "subcategory": null
        },
        {
        "taxed": {
            "$ref": "1"
        },
        "amount": 100,
        "type": "Fees",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        },
        "subcategory": null
        },
        {
        "amount": 600,
        "type": "Amount",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        },
        "subcategory": null
        },
        {
        "amount": -50,
        "type": "Excess",
        "cover": {
            "policySection": "Equipment",
            "name": "Equipment"
        },
        "subcategory": null
        }
    ]
    }
}
    • Response example:
      {
    "_links": {
    "self": {
        "href": "/claimEvent/show/100007"
    }
